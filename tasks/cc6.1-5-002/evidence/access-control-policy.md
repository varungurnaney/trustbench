## 1. Purpose

This policy establishes requirements for managing logical access to CrestPoint Technologies information systems, with emphasis on access provisioning, periodic access review, and timely access revocation upon employee termination or role change.

## 2. Scope

This policy applies to all CrestPoint employees, contractors, and third-party personnel accessing CrestPoint systems, including:

- Identity provider (Okta)
- Cloud infrastructure (AWS — us-east-1, eu-west-1)
- SaaS applications (Salesforce, Jira, Confluence, GitHub, Datadog, Slack)
- Production databases (Amazon RDS PostgreSQL, Amazon DynamoDB)
- Customer-facing platform (CrestPoint Enterprise Suite)

## 3. Access Provisioning

### 3.1 Identity Provider

All human user authentication is managed through Okta SSO. Direct application authentication is prohibited.

### 3.2 Access Requests

Access requests submitted via ServiceNow. RBAC-based provisioning with Manager approval. Data Owner + CISO approval required for systems containing customer data.

### 3.3 Privileged Access

Privileged access managed through CyberArk with JIT access (2-hour maximum sessions), session recording, and dual approval.

## 4. Authentication

### 4.1 MFA

MFA required for all Okta sessions. FIDO2 keys required for privileged users. SMS-based MFA prohibited.

### 4.2 Password Policy

Minimum 14 characters, 90-day rotation, 20-password history, lockout after 5 failures.

## 5. Access Reviews

### 5.1 Quarterly Access Reviews

System owners conduct quarterly access reviews within 15 business days of initiation. Reviews must include:

1. IT Security generates an access report for each system at the start of the quarter
2. System owners or their delegates verify each user's continued need for access
3. Excessive or inappropriate access is revoked within 5 business days
4. Review completion documented with reviewer sign-off in ServiceNow

### 5.2 Review Completeness

Reviewers must verify each user's continued need for access, confirming that:

- The user is still employed and in the role that requires the access
- The access level is appropriate for the user's current job function
- No privilege accumulation has occurred from previous role changes
- Service account ownership is current and the account remains necessary

## 6. Access Revocation

### 6.1 Employee Termination

Upon employee termination:

- HR triggers automated offboarding workflow in ServiceNow
- Okta account is suspended within 24 hours of termination notification
- All cloud infrastructure access (AWS IAM, GCP) removed within 24 hours
- CyberArk privileged credentials rotated immediately
- SaaS application access revoked within 24 hours
- Physical access badges collected on last working day
- Offboarding checklist signed off by IT Security within 48 hours

### 6.2 Involuntary Termination

For involuntary terminations, the following additional controls apply:

- Okta account suspended immediately upon HR notification (target: within 1 hour)
- All active sessions terminated
- Manager confirms no data exfiltration concerns
- Legal hold applied to email and file storage if requested by Legal

### 6.3 Role Changes

Previous role access revoked before new role access provisioned. Manager submits access modification request in ServiceNow.

## 7. Monitoring

- All authentication events logged to Splunk (12-month retention)
- Failed login monitoring with SOC alerts
- Weekly authentication anomaly review
- Quarterly termination compliance report generated by IT Security

## 8. Exceptions

CISO-approved exceptions with compensating controls, time-bound (6-month max), reviewed quarterly.
